## Interface Management - CRITICAL

### Hierarchy System
- **Hierarchy files (.hier)**: Define parent-child relationships
- **Interface files (.icf)**: Define signal routing between modules  
- **Generated ports**: Auto-generated in `target/{cluster}/gen/` - **NEVER manually modify**

### Signal Flow Pattern
```
child_module (generates) → container.icf (routes) → sibling_module (receives)
```

### ICF Declaration Process
```systemverilog
// In source container .icf
output node [WIDTH-1:0] SignalNameM401H
    pin {source_module} output;

// In destination container .icf  
input node [WIDTH-1:0] SignalNameM401H
    pin {dest_module} input;

// Between siblings in same container
node [WIDTH-1:0] SignalNameM401H
    pin {source_module} output
    pin {dest_module} input;
```

### Critical Steps
1. **Identify usage**: Use `grep_search` to find which modules actually use the signal
2. **Add to ICF**: Only add as input to modules that reference it in RTL
3. **Remove internal nodes**: Delete `node` declarations from RTL after converting to interface
4. **Match array format**: Packed `[WIDTH-1:0] Signal` vs Unpacked `Signal [WIDTH-1:0]`

### Signal Naming Conventions
- **MnnnH**: Generic pipeline stage (nnn = stage number)
- **M40XH**: Specific pipeline stages (X = 0-9)
- **M4nnH**: 400-series pipeline stages for critical timing paths

### Interface Files (.icf) and Hierarchy Architecture
- *.icf: Defines all input/output signals for the specified module
- Interface signals must be consistently defined as input/output across modules

#### Hierarchical Interface Architecture - CRITICAL
Intel uses a hierarchical interface system:

1. **Hierarchy Files (.hier)**: 
   - `mob.hier` describes top-level hierarchy with child modules
   - Each container module has its own .hier file describing its children

2. **Interface Files (.icf) Usage**:
   - `.icf` files define interfaces for containers with multiple child modules
   - Use parent container's .icf to define signals between sibling modules
   - Example: Use `mob.icf` to connect signals between `moscb` and `mosab` (both children of mob)

3. **Generated Port Files**:
   - `*port.v` files in `target/<cluster>/gen/` are auto-generated by IronChef
   - **NEVER manually modify** these generated files
   - All port changes must be made through .icf files
   **Good Usage:**
   ```systemverilog
   // In .icf file
   output node [WIDTH-1:0] NewSignalM200H
       pin {module_name} output;
   ```

   **Bad Usage:**
   ```systemverilog
   // Manually editing module port list in .vs file
   module my_module (
       input node NewSignalM200H // DO NOT ADD MANUALLY
   );

#### Signal Flow Through Hierarchy Example:
```
moscb_merge_ctrl (child generates signal) 
    → moscb.icf (defines output from child)
        → moscb container (outputs to parent)  
            → mob.icf (routes between siblings)
                → mosab (receives as input)
```

#### Interface Files (.icf) - CRITICAL
Interface files define signals between modules in {container}.icf:
```systemverilog
// Output declaration in source container module .icf
output node SignalNameM404H[`SIGNAL_WIDTH]
    pin {source_module} output
;

// Input declaration in destination container module .icf  
input node SignalNameM404H[`SIGNAL_WIDTH]
    pin {target_module} input
;

// Declaration for interface between modules inside same container module .icf
node SignalNameM404H[`SIGNAL_WIDTH]
    pin {source_module} output // Source module output
    pin {target_module} input   // Target module input
;

```

### Interface Signal Management Best Practices
When adding signals to ICF files as inputs/outputs for modules:

1. **Identify Signal Usage**: 
   - Use `grep_search` to find which modules actually use the signal
   - Only add signal as input to modules that reference it in their RTL code

2. **Proper ICF Declaration Process**:
   ```systemverilog
   // In source module .icf (e.g., mob.icf) - where signal is generated
   output node [DC_NUM_SLICES-1:0] SignalNameM401H
       pin {module_name} output
   ;
   
   // In destination module .icf (e.g., moglbd.icf) - where signal is used
   input node [DC_NUM_SLICES-1:0] SignalNameM401H
       pin {module_name} input 
   ;
   ```

3. **Remove Module Declarations**:
   - **CRITICAL**: After adding signal to ICF files, remove the `node` declaration from the RTL module
   - Example: Remove `node [DC_NUM_SLICES-1:0] SignalNameM401H;` from the module's .vs file
   - The ICF interface system now manages the signal connection
   - **IMPORTANT**: Only remove the specific signal being converted - preserve all other node declarations

4. **Safe Node Declaration Removal**:
   ```systemverilog
   // BEFORE: Internal node declaration
   node SBTailWrEnMnnnH;
   node AtRetDealM391H;
   node AtRetDealM392H;
   
   // AFTER: Remove only the converted signal, keep others
   // (SBTailWrEnMnnnH removed - now interface signal)
   node AtRetDealM391H;
   node AtRetDealM392H;
   ```
   - Always verify that other signals are still used before removing
   - Use `grep_search` to check signal usage throughout the module

5. **Signal Routing Pattern**:
   - Signal flows: `source_module` (generates) → `destination_module` (uses as input)
   - Only add as input to modules that actually reference the signal in their logic
   - Check bit widths and array dimensions

6. **Interface Array Declaration Compatibility**:
   - **CRITICAL**: Packed vs unpacked array declarations must match exactly across container hierarchy
   - **Packed**: `input node [DC_NUM_SLICES-1:0] SignalName` (dimension before signal name)
   - **Unpacked**: `input node SignalName [DC_NUM_SLICES-1:0]` (dimension after signal name)
   - Use `grep_search` to verify existing declaration format in parent containers (e.g., mob.icf)
   - Common error: "Cannot connect packed to unpacked array" in ironchef compilation

7. **Bit Width Verification**:
   - Verify width compatibility when connecting signals between modules


#### **Key Principles for Instrumental Signals**:

1. **Purpose**: Create reference implementations to verify functional correctness of optimized silicon logic

2. **Naming Convention**: Add `_inst` suffix to the original signal name
   ```systemverilog
   // Functional signal
   t_SBID SBRetPtrMnn1H;
   
   // Instrumental version  
   t_SBID SBRetPtrMnn1H_inst;
   ```

3. **Signal Declaration**: Place within `INTEL_INST_ON` ifdef blocks
   ```systemverilog
   `ifdef INTEL_INST_ON
      t_SBID SBRetPtrMnn1H_inst;
      t_SBID SBRetPtrMnnnH_inst;
   `endif
   ```
