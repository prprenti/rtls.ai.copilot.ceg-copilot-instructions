# IP Release Reference

## Safety and Scope

- `ip_release` is a destructive, irreversible operation that creates a new
    immutable FQN version.
- All uploads must go through `ip_release`; do not use raw
    `ipx ip upload`, `ipx ip create`, or `ipx ip branch` for release flows.
- Incorrect Unix group, permissions, Carbon ID, or lineage parent can create
    tracking and compliance issues.

If there is uncertainty about release lineage or target configuration, escalate
to the owning release experts before execution.

## Required Confirmation Protocol

Before executing `ip_release`, the assistant must:

1. Show the exact command.
2. Explain target library/IP, milestone, and lineage parent (`-previous_fqn`).
3. Explicitly confirm `-previous_fqn` with the user.
4. Wait for explicit user approval before execution.

## Required Parameters

| Parameter | Description |
|---|---|
| `-file DIR` or `-tar TARBALL` | Path to directory or tarball to upload (use one) |
| `-fqn <FQN>` | New FQN to create (e.g., `nvlhaxfe.sncu@.ww50_ev1` for first upload) |
| `-carbon_ip_config <ID>` | Carbon Config ID. Use `-carbon_ip_config_2 <ID>` for new project branch mismatches |
| `-milestone <milestone>` | Milestone tag: `EV1`, `EV2`, `UV1`, `UV2`, `FV1`, `FV2`, etc. |
| `-list_of_hsds <file>` | Text file with HSD-ES bug-eco numbers, one per line |
| `-ipx_site zsc10` | IPX site to upload to |
| `-previous_fqn <FQN>` | FQN of the immediately prior drop. Use `TRUNK` for first drop on new line |
| `-ship` | Mark IP as shipped/released |
| `-keep_unsafe_link` | Allow symlinks to paths outside the repo |

## Prerequisites Detail

### CTH Setup Window (REQUIRED)

`ip_release` must run within a CTH setup window.

Use the FE setup MCP readiness method directly:

```text
fe-setup/check_terminal_ready()
```

Proceed only when this returns `true`. If it returns `false`, run FE setup
first, then re-check readiness before continuing.

All commands in the sections below (starting with IPX client setup) are
post-readiness steps and must run only after `check_terminal_ready()` passes.

### IPX Client Setup (REQUIRED)

```bash
source /p/ipx/cad/ipxclient/prod/setup/ph3_ipxclient.setup
```

### Carbon Config ID

Every upload requires a Carbon Config ID (`-carbon_ip_config`) tying the IP
to a project in the Carbon BOM tracking system.

**Where to find it:**
- `handoff/ipqc/flow.cfg` → `IP_CONFIG_ID`
- Carbon BOM tree (user needs to open manually and do a search): https://hsdes.intel.com/appstore/carbon/bom/tree/basic/22018048851

> **IMPORTANT:** Different dies may have different Carbon IDs. New project
> branches (A0 vs B0) may produce new IDs. If `-carbon_ip_config` causes a
> mismatch, use `-carbon_ip_config_2 <ID>`.

### Determine the FQN

- **Library name:** from `git config --get intel.stepping`, take portion before first dash
- **IP name:** from `git config --get intel.cluster`
- **Example:** if stepping = `ddgip-trunk`, cluster = `sncu` → `ddgip.sncu@.ww50_ev1`

### Drop Size Verification

Goal: keep a typical drop around **1-2 GB**. If the drop grows well beyond
this range, validate package composition before upload.

Use these checks before `ip_release`:

```bash
# Tarball
tar tzvf output/sncu.fe_package.tar.gz | awk '{s+=$3} END{print (s/1024/1024/1024), "GB"}'

# Directory
du -sh output/package/fe_package_untar/

# Optional: find largest subdirectories/components
du -h output/package/fe_package_untar/ | sort -h | tail -30
```

If size is high, prune non-reuse payload before packaging:
- Remove flattened sub-IP content that is not consumed by downstream reuse.
- Remove duplicate/generated artifacts that can be regenerated by consumers.
- Keep only the minimal deliverables required by integration and signoff.
- Some repos already provide a pruning helper in their packaging flow area.
    If present, run that repo-provided pruning script before upload when the
    drop is excessively large.

Re-run size checks after pruning and confirm final payload is in the expected
range before upload.

If the final drop size still exceeds **2 GB**, require explicit user approval
before running `ip_release`. The user should clearly acknowledge that a larger
than normal payload is acceptable for this upload.

### HSD Bug-Eco List

Plain text file with one HSD-ES article number per line:

## Command Template

```bash
ip_release <-file DIR | -tar TARBALL> \
    -keep_unsafe_link \
    -ship \
    -fqn <new_FQN> \
    -carbon_ip_config <carbon_id> \
    -milestone <milestone> \
    -list_of_hsds <bugeco_file> \
    -ipx_site zsc10 \
    -previous_fqn <previous_FQN>
```

## IP Lineage — Detail

### How Lineage Works

Each drop records its parent via `-previous_fqn`, creating a chain:

```
TRUNK → drop_A → drop_B → drop_C → drop_D
```

If drop A fixes 3 bugs and drop B fixes 8 bugs, then drop C automatically
shows fixes for A + B + its own. Wrong `previous_fqn` breaks this chain.

### Linear Drops (Common Case)

| New FQN | Previous FQN |
|---|---|
| `nvlhaxfe.sncu@.ww50_dropA` | `TRUNK` (first drop) |
| `nvlhaxfe.sncu@.ww50_dropB` | `nvlhaxfe.sncu@1.ww50_dropA` |
| `nvlhaxfe.sncu@.ww50_dropC` | `nvlhaxfe.sncu@1.ww50_dropB` |
| `nvlhaxfe.sncu@.ww50_dropD` | `nvlhaxfe.sncu@1.ww50_dropC` |

> Use meaningful drop names: `ww50_ev1`, `ww03_uv1` — not `A`, `B`, `C`.

### Non-Linear Drops (Branching)

When fixing an older branch while newer branches exist:

```
TRUNK → A0_drop1 → A0_drop2
                 ↘ B0_drop1 → B0_drop2
                            ↗
              A0_drop2_fix1 (previous_fqn = A0_drop2, NOT B0_drop2)
```

### Verifying Lineage

```bash
ipx ip lineage -ip nvlhaxfe.sncu@3.ww50_dropC -ancestors
```

**If there are ANY questions about lineage, contact Greg Ruhl or FEMI.**

## How to Determine Required Fields

### Library Name

From `git config --get intel.stepping`, take portion before first dash.

- `ddgip-trunk` → library = `ddgip`
- `nvlhaxfe-a0` → library = `nvlhaxfe`

### IP Name

From `git config --get intel.cluster`.

- `sncu` → IP name = `sncu`

### FQN Construction

Combine library + IP name:

- New branch: `ddgip.sncu@.ww50_ev1`
- Specific version: `ddgip.sncu@3.ww50_ev1`

### Milestone

Must be specified by the user. AI should prompt:

> "What milestone should this drop be tagged with? Options: `EV1`, `EV2`,
> `UV1`, `UV2`, `FV1`, `FV2`, etc."

Reuse of the same milestone across drops is OK.

### Drop Path

Check `ip_release_config.json` for the configured `drop_path`, or the user
must specify in chat.

### Previous FQN

Must be provided by the user or looked up via `ipx ip lineage`. If not
specified, the AI must ask and verify before proceeding.

## Common Mistakes

| Mistake | Consequence | Prevention |
|---|---|---|
| Wrong `-previous_fqn` | Broken lineage | Verify with `ipx ip lineage` after upload |
| Wrong `-carbon_ip_config` | Carbon BOM mismatch | Check `flow.cfg` or `ip_release_config.json` |
| Missing `-keep_unsafe_link` | Symlinks rejected | Always include unless no external symlinks |
| Drop too large (tens of GBs) | Upload timeout | Check size before uploading |
| Using same milestone incorrectly | Confuses consumers | Use descriptive drop names |
| Outside CTH setup window | `ip_release` not found | Run setup per copilot-instructions.md |
| ddgip symlinks to Hub/Cdie | Symlinks not visible at target | Verify symlink targets are accessible |

## Post-Upload Verification

Run these checks immediately after upload:

```bash
# 1) Verify lineage chain
ipx ip lineage -ip <new_FQN> -ancestors

# 2) Compare content against prior release (recommended)
ipx ip compare -reference <previous_FQN> -compare <new_FQN> -file -bom

# 3) Sync to remote sites if required
ipx ip sitesync -ips <new_FQN> -sites sc fm -wait
```

If lineage output does not reflect the expected parent chain, stop and correct
before downstream consumers adopt the release.
