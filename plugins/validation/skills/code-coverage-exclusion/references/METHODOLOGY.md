# Code Coverage Exclusion - Detailed Methodology

## Step-by-Step Workflow

### Step 1: Gather Information

**Required files:**
1. **Sample .el file** - Extract INSTANCE path and format template
2. **RTL module** - Port declarations to scan
3. **TB connection file** - Identify strapped signals

### Step 2: Define Exclusion Rules

```python
patterns = [
    r'.*fscan.*',     # Scan signals
    r'.*ijtag.*',     # JTAG test access
    r'^fp_.*',        # Floorplan configs
    r'.*debug.*',     # Debug infrastructure
    r'.*test_mode.*', # Test modes
]
```

### Step 3: Scan RTL for Pattern Matches

1. Parse RTL module port declarations (input/output only)
2. For each port signal, check against exclusion patterns
3. If match, capture signal name and type declaration
4. **Skip internal wires/logic** - only interface signals

```bash
grep -E "^\s*(input|output)" <rtl_file> | grep -E "(fscan|ijtag|^fp_)"
```

### Step 4: Scan TB for Strap Connections

1. Find module instantiation in testbench
2. Parse port connections: `.<port>(value)`
3. Identify constant assignments

**Constant detection heuristics:**
- Starts with digit: `0`, `1`, `16'h`, `8'd`
- Contains `::` (package scope): `PKG::CONST`
- Single quote literals: `'0`, `'1`
- All uppercase: `MY_CONSTANT`

### Step 5: Generate Exclusion File

```verilog
//==================================================
// This file contains the Excluded objects
// Generated By User: <username>
// Format Version: 2
// Date: <timestamp>
// ExclMode: default
//==================================================
CHECKSUM: "0 0"
INSTANCE: tb.top.module_inst
Toggle scan_signal_1 "input logic"
Toggle scan_signal_2 "input logic [7:0]"
Toggle strapped_config "input logic [15:0]"
```

**Rules:**
- Sort entries alphabetically by signal name
- One Toggle entry per excluded signal
- Format: `Toggle <signal_name> "<signal_type>"`

### Step 6: Validation

| Check | Action |
|-------|--------|
| Duplicates | Remove duplicate signal entries |
| RTL existence | Verify signals exist in RTL module |
| Type match | Confirm signal types match RTL declarations |
| INSTANCE path | Validate path matches TB hierarchy |

## Error Handling

| Issue | Cause | Solution |
|-------|-------|----------|
| No pattern matches | Patterns too restrictive | Review patterns, check RTL |
| Duplicate exclusions | Signal matched both checks | De-duplicate before output |
| Signal type mismatch | RTL parsing error | Verify signal declarations |
| Invalid INSTANCE path | Wrong hierarchy | Update from sample .el |
| Signal not found | Stale TB or typo | Flag warning, skip entry |

## Output File Naming

```
<module_name>_exclusion.el
```

Examples: `DE_exclusion.el`, `NOC_exclusion.el`, `IOSF_exclusion.el`

## Post-Generation

1. **Review** - Verify key exclusions match expectations
2. **Integrate** - Place in coverage directory:
   ```
   src/val/coverage/<stepping>/iu.<module>/exclude/<module>_exclusion.el
   ```
3. **Validate** - Run coverage tool to verify syntax
